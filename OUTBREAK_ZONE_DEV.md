## 프로젝트 개요
- **게임명:** OUTBREAK ZONE
- **장르:** 2D 탑다운 서바이버 퍼즐 (지뢰찾기 + 좀비 + 로그라이크)
- **플랫폼:** 웹 (CrazyGames) → Android Google Play
- **개발 방식:** HTML5 Canvas + Vanilla JS

## 작업 원칙

### 코드 수정 원칙
- 수정 시 항상 쓰레기코드 제거
- 다른 코드와 연결된 경우 전체 파악 후 수정
- 가능하면 문자열 교체보다 전면 재작성 방식 권장
- 수정 후 잔재 코드 검증 필수 (grep/count 확인)

### 커밋 메시지 방식
```
한글로 짧게
예시:
  스탭3 - 좀비 스폰, CHASE 이동, 접촉 피해
  스탭3 패치2 - 시야LOS, 소음인식, 분리흩어짐, 스폰분산
  스탭3 정리 - 쓰레기코드 제거 (DIR8, placed, _sonarWarmCtx)
```

### 피드백 방식
- 코드 수정 전 "코드수정없이 피드백먼저" 요청 시 피드백만
- 검증 요청 시 치명적 문제 아니면 '가능합니다'로 처리 후 수정
- 방향성 파악 후 세부 피드백 없이 바로 수정 가능
- 피드백 내용을 길게 나열하지 않고 핵심만

### RESOURCE LAYER
- 렌더링 함수 (drawXxx)는 RESOURCE LAYER로 분리
- 스프라이트 교체 시 해당 함수만 수정
- 모든 draw 함수는 ctx.translate(-camX,-camY) 적용 상태에서 호출
- **월드 좌표 그대로 사용 (camX/Y 빼지 않음)**

## 개발 단계 현황

### ✅ Step 1 — 봉인 완료
- 미로 맵 생성 (재귀 백트래커 + 루프 경로)
- WASD 이동 (타일 단위, 4방향)
- 3단계 시야 (완전밝음 / 그림자 / 암흑)
- 카메라 추적 (smooth)
- 미니맵 (방문한 곳만 표시, 80px 우하단)
- facing: 이동 불가여도 방향 즉시 업데이트
- 막다른 길 비율 체크 (30% 초과 시 재생성)

### ✅ Step 2-A — 봉인 완료
- 지뢰 배치 (스폰 주변 2타일 안전 반경)
- 지뢰 밟으면 HP -1 + 빨간 플래시
- HP 하트 아이콘 연동
- 게임오버 화면 + RETRY

### ✅ Step 2-B — 봉인 완료
- **기본 소나 [F키 홀딩]:** 지뢰 인접 타일에 위험도 핑 (노랑/주황/빨강)
- **정밀 소나 [G키 홀딩]:** 지뢰 타일에 X 표시 (횟수 소모)
- 소나 파장 애니메이션 (순차 점등)
- 핑 N초 후 페이드아웃
- 밟은 지뢰 제거 후 numbers 재계산
- HUD: 기본소나[F] / 정밀소나[G] 📡 카운터

### ✅ Step 2-C — 봉인 완료
- 아이템 타일 (탈출 부품 🔑) — 거리 구간 분산 배치
- 출구 타일 (🚪) — 스폰 최원거리 배치
- 탈출 조건 (N개 수집 → 출구 개방)
- 탈출 성공 화면 (#escaped)
- 아이템 수집 피드백: 초록 vignette + HUD 숫자 팝 (지뢰 빨강 플래시와 구분)
- 미니맵: 아이템 노란 점 / 출구 잠금(어두움)→해제(밝은 초록)

### ✅ Step 3 — 봉인 완료
- 좀비 엔티티 (픽셀 단위 이동, 벽 슬라이딩 충돌)
- **상태 머신:** WANDER / SEARCH / CHASE
  - WANDER: 랜덤 배회 (1.5~3.5초마다 방향 전환, 벽 유효 방향 우선)
  - SEARCH: 시야 잃으면 마지막 목격 위치로 이동 → 두리번 → 2초 후 WANDER
  - CHASE: 플레이어 추적
- **인식 3종:** 시야(FOV 각도 + LOS), 소음(noiseRadius), 근접(0.6타일)
- 좀비 시야 벽 관통 차단 (hasLOS 활용)
- 좀비 간 분리 벡터 (겹침 방지 + 흩어짐)
- 스폰: 거리 구간 분산 배치
- 접촉 피해 (player.damageCooldown 통합 관리)
- 암흑 속 좀비 미렌더링 (VISIBLE 체크)

#### Step 3 최적화/리팩토링 이력
- updateZombies 157줄 → 35줄 (상태별 함수 분리)
  - zombieSense / zombieMove(zombieChase/Search/Wander) / zombieSeparate / zombieApplyMove / zombieContact
- 매 프레임 공유 컨텍스트 객체(ctx2)로 전달
- 쓰레기코드 제거: DIR8, placed 배열, _sonarWarmCtx defineProperty

#### Step 3 난이도 너프 이력
- 지뢰 밀도 15% → 8%
- 좀비 수 3 → 2마리
- 근접 감지 1.2 → 0.6타일
- SEARCH 시간 3.5 → 2.0초
- 소나 소음 반경×1.5 → ×0.8, 지뢰 소음 8 → 5타일

## 플레이어 행동 → 소음(noiseRadius) 시스템
| 행동 | noiseRadius (타일) |
|------|------|
| 이동 | 0 (소리 없음) |
| 소나 발동 | 스캔 반경 × 0.8 |
| 지뢰 밟음 | 5 |

- noiseRadius는 매 프레임 `dt * 3`씩 감소
- 좀비는 noiseRadius 범위 안에 있으면 CHASE 전환 (벽 관통 인식)

## 프로젝트 방향성 진단 (Step 3 봉인 시점)

### 핵심 컨셉 평가
**"멈춰서 정보를 캐야 하는데(지뢰), 멈추면 위협이 다가온다(좀비)"** — 이 긴장이 성립하면 게임이 됩니다.

소나가 이 긴장의 정중앙에 있습니다. 쓰면 지뢰 정보를 얻지만 소음으로 좀비를 부른다 — 하나의 행동에 리스크/리워드가 묶여 있어요. 기획적으로 매우 단단한 코어입니다.

---

### 기획자 관점

**장점**
- 코어 루프가 명확하고 확장 여지가 큼. 직업/감염/동료가 전부 소나-긴장 위에 자연스럽게 얹힙니다.
- config로 파라미터가 완전 분리돼 밸런싱 반복이 빠름. 이런 장르는 수십 번 수치를 만져야 완성되는데 구조가 잘 잡혀 있습니다.
- RESOURCE LAYER 분리로 아트 교체가 코드와 독립적. 프로그래머 아트로 코어 검증 후 스프라이트 입히는 정석 순서를 밟고 있습니다.

**우려**
- **난이도 곡선을 아직 손에 못 쥐고 있음.** "좀비가 너무 똑똑하다 → 너프했더니 너무 멍청하다"를 반복한 건, 플레이어의 **대항 수단이 없기** 때문입니다. 회피밖에 없으면 좀비가 조금만 영리해도 불공평하고 조금만 둔하면 무의미해집니다. Step 4의 낚시 메카닉이 이걸 풀 열쇠 — 단순한 다음 기능이 아니라 **밸런스의 잃어버린 절반**입니다.
- **승리 조건이 단조로움.** "아이템 3개 줍고 출구로"는 의사결정 다양성이 적습니다. 로그라이크의 "이번 판은 다르다"가 아직 약합니다.

---

### 소비자 관점

**처음 60초:** 어두운 미로 + 두근거리는 소나 + 갑자기 튀어나오는 좀비. 첫인상과 무드는 잡혀 있습니다.

**3분 차:** 좀비를 "다룰 수 있다"고 느끼면 몰입, "그냥 운"이라고 느끼면 이탈. 현재는 후자에 가까운 위험이 있습니다. 복도 마주침이 피할 수 없다는 피드백이 정확히 이 지점.

**리텐션:** CrazyGames/모바일은 즉각적 보상과 짧은 세션이 생명. 죽으면 0으로 돌아가는 구조에서 "지고도 남는 것"(점수, 베스트 기록, 클리어 횟수)이 생각보다 일찍 필요할 수 있습니다. 현재 이 요소가 Step 6에 몰려 있음.

---

### 발전 가능성

상한선은 높습니다. "지뢰찾기를 실시간 생존으로"는 한 줄 피치가 명확하고 유니크합니다. 스트리밍/숏폼에서 클립이 잘 나오는 종류의 게임이에요.

### 권장 우선순위

**1. Step 4 낚시 메카닉 최우선**
소나 발원지로 좀비가 반응하는 것은 콘텐츠가 아니라 밸런스의 누락분. 플레이어에게 좀비를 다룰 능동적 수단이 생기는 순간 난이도 튜닝이 비로소 의미를 가집니다.

**2. "한 판"의 최소 완결성 조기 추가**
베스트 기록 또는 클리어 횟수 하나만 일찍 넣어도 리텐션 체감이 크게 달라집니다.

**3. 외부 플레이테스트**
개발자는 시스템을 다 알아서 객관적 난이도 판단이 어렵습니다. 아무것도 모르는 사람 2~3명에게 60초 쥐여주고 어디서 막히는지 보는 게, 수치 100번 만지는 것보다 빠릅니다.
